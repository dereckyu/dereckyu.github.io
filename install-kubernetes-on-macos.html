这里需要保持上面 log 结尾处 kubeadm join这行指令，后续需要用它部署 Worker 节点。

另外，需要根据 log 中的提示拷贝集群所需配置到 user home 目录下。而需要这些配置命令的原因是：Kubernetes 集群默认需要加密方式访问。所以，这几条命令，就是将刚刚部署生成的 Kubernetes 集群的安全配置文件，保存到当前用户的.kube 目录下，kubectl 默认会使用这个目录下的授权信息访问 Kubernetes 集群。

查看 master 节点状态，发现其状态是 NotReady

这时因为网络插件尚未部署 “message:docker: network plugin is not ready: cni config uninitialized”

除了通过 kubectl describe 查看原因，还可以通过查看节点上的所有 Pod 的状态来分析。其中，kube-system 是 Kubernetes 项目预留的系统 Pod 的工作空间（Namepsace，注意它并不是 Linux Namespace，它只是 Kubernetes 划分不同工作空间的单位）：

可以看到，依赖于网络的 CoreDNS pod 调度失败，一直在 Pending。

本地访问

由于 Dashboard 是一个 Web Server，很多人经常会在自己的公有云上无意地暴露 Dashboard 的端口，从而造成安全隐患。所以，1.7 版本之后的 Dashboard 项目部署完成后，默认只能通过 Proxy 的方式在本地访问。

这里在 master 节点运行 kubectrl proxy 后，就可以在 master 节点内通过 http://localhost:8001/api/v1/namespaces/kubernetes-dashboard/services/https:kubernetes-dashboard:/proxy/ 访问 Dashboard 了

使用上一步生成的 token 登录，可看到集群的具体信息

部署容器存储插件 - Rook (可选)

如果只是在本地测试一下 k8s 集群，那无需此插件。这个插件主要解决不同机器上的容器需要共享容器数据的场景。比如，我们需要搭建一个 redis 集群，redis 的各节点（这里以 Pod 方式运行）需要持久化数据（rdb、aof），这样当 redis 节点 crash 之后依旧有能力恢复数据。此时就需要有个存储插件来保持数据文件，比如 NAS。
